- **Пропустить:** флаги `[processed]`, `[duplicate]`, `[rejected]` (уже закрыты), а также заметки с конкретным Домен/Тип (уже классифицированы)
- Если новых/необработанных заметок нет → сообщить «Новых заметок нет» и завершить

#### 2. Прочитать контекст

- `DS-strategy/docs/Dissatisfactions.md` — текущие НЭП (S* стратегические, O* операционные)
- `DS-strategy/current/WeekPlan W*.md` — РП и приоритеты недели

Замечания (✏️) и urgent (high/critical) — в начале файла. Real-time алерты уже отправлены ботом → Note-Review только проверяет кластеры (≥3 в одном кластере → пометить)

#### 3. Классифицировать каждую заметку

| # | Категория | Критерий | Куда пишет Note-Review |
|---|-----------|----------|------------------------|
| 1 | **НЭП** | Неудовлетворённость, дискомфорт, «хочу изменить», эмоция | → предложение в конец Dissatisfactions.md |
| 2 | **Задача** | Конкретное действие, «завтра сделать», «нужно проверить» | → предложение в конец WeekPlan |
| 3 | **Знание** | Паттерн, различение, метод, правило, инсайт | → запись в DS-strategy/inbox/captures.md |
| 4 | **Черновик** | Зерно для поста/статьи, развёрнутая мысль, рефлексия с концептами | → рекомендация в конец WeekPlan (секция «Предложения Note-Review»). Файл НЕ создаётся — только после согласования |
| 5 | **Идея** 🔄 | Размышление, требует осмысления, нет конкретного действия | → пометить 🔄 в fleeting-notes.md |
| 6 | **Личные данные** | Контакт, телефон, email, токен, аккаунт, реквизиты, справочные данные | → DS-strategy/personal/*.md |
| 7 | **Бот-замечание** | Баг, UX-проблема, пожелание к боту (слова: бот, /команда, кнопка, урок, глючит, не работает, ошибка в боте, лента, дайджест) | → предложение в DS-strategy/inbox/WP-7-bot-tech-debt.md |
| 8 | **Шум** | Тест, дубль, случайное нажатие, ссылка без контекста, уже сделано / решено | → зачеркнуть ~~текст~~ |

**Правила классификации:**
- Одна заметка = одна категория (главная)
- Если НЭП + задача → приоритет НЭП (эмоция важнее действия)
- Если бот + задача → приоритет **Бот-замечание** (маршрутизация в WP-7 важнее общей задачи)
- **Черновик vs Идея:** Заметка работает с концептами, содержит рассуждение, ссылается на прочитанное/изученное → **Черновик** (зерно поста). Заметка абстрактная, нет связи с концептами → **Идея** 🔄
- **Черновик vs Знание:** Знание = факт/правило/различение для Pack (формализация). Черновик = связный текст для поста (публикация). Одна заметка может породить и то, и другое — тогда записать в ОБА места
- Длинные рефлексии без дискомфорта и без концептов → Идея 🔄
- Ссылки без контекста → Шум
- **Срочные задачи:** Если заметка содержит срок/дедлайн («сегодня», «завтра», конкретную дату) → категория **Задача** с пометкой срока в предложении. Предложение пишется в DayPlan (если срок = завтра) или WeekPlan (если срок = на неделе). Если срок уже прошёл — всё равно записать как просроченное
- **Личные данные → personal/:** Контакт/телефон → `personal/contacts.md`. Аккаунт/логин/URL → `personal/accounts.md`. Токен/ключ/пароль → `personal/secrets.md`. Прочие справочные данные → `personal/reference.md`

**Проверка актуальности (обязательно перед классификацией):**
- Сверь заметку с коммитами за сегодня (`git log --since="00:00" --all --oneline` по всем {{WORKSPACE_DIR}}/ репо) и текущим WeekPlan
- Если задача из заметки **уже сделана** (есть коммит или РП done) → Шум (зачеркнуть), не тянуть в план
- Если проблема из заметки **уже решена** → Шум
- Если заметка ссылается на ситуацию, которая **изменилась** за день → оценить с учётом нового контекста
- Пользователь мог написать утром, а к вечеру всё уже сделал — не создавай лишних предложений
- **Сопоставление с РП (для ВСЕХ категорий, включая Идею 🔄):** Перед классификацией любой заметки — проверь по `memory/MEMORY.md`: совпадает ли intent заметки с существующим РП? Проверяй НЕ только название РП, но и описание результатов (✅ строки). Пример: заметка «Сделать стандарт описания роли» покрыта РП #54 (DP.D.033 Role-Centric Architecture) — даже если РП in_progress, конкретный результат done. Если покрыто → Шум (зачеркнуть, указать «Реализовано: РП #N, артефакт»), НЕ ставить 🔄. Правило: **заметка, чей intent уже реализован = Шум, независимо от статуса РП**
- **Личные задачи:** Заметки с конкретным действием и сроком (в т.ч. личные, бытовые, семейные) → категория **Задача**. Записывать в WeekPlan с дедлайном. НЕ отбрасывать как «нерелевантные». Недельный план включает ВСЕ задачи: рабочие + личные. Нумерация личных: Л1, Л2, ...

## Автоматическая конвертация "Задача" → WP (Note-to-WP pipeline)

> **НОВОЕ (ENG.WP.047):** После Note-Review завершена классификация, все "Задача" должны автоматически конвертироваться в работные продукты (РП).

### Когда

**Оптимальное время:** сразу после завершения Note-Review (в той же сессии или автоматически ночью через scheduler).

Триггер: Note-Review обработал заметки и написал предложения "Задача" в WeekPlan → **автоматически вызвать note-to-wp.md**.

### Как

Для каждого предложения "Задача" в конце текущего WeekPlan:

```bash
/note-to-wp
```

(или в автоматизированном режиме: scheduler запускает note-to-wp.md с параметром `--auto-process-all-tasks`)

### Результат

Каждая классифицированная "Задача" → **новый WP-файл в inbox/** → регистрация в 5 местах (REGISTRY, WeekPlan, MEMORY, fleeting-notes, marker в предложении).

**Статус после note-to-wp:**
- ✅ WP-файл создан и готов к выполнению
- ✅ Видится в inbox/ и будет показан в Day-Open
