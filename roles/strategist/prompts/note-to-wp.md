Выполни сценарий «Конвертация классифицированной заметки в рабочный продукт» для роли Стратег (R1).

## Контекст

Это промпт **автоматической конвертации классификации "Задача" → WP**.

Триггер: Note-Review определил заметку как "Задача" и написал предложение в конец WeekPlan.
Результат: Новый файл WP-NN-*.md в {{WORKSPACE_DIR}}/inbox/, регистрация в 5 местах.

## Входные данные

1. **Предложение из WeekPlan:** строка от Note-Review вида:
   ```
   > **Note-Review:** [время] Классификация: Задача
   > Текст: «{заголовок}»
   > Контекст: {область, связь}
   > Параметры (нужно определить): name, area, repo, budget, criteria
   ```

2. **Исходная заметка:** из `{{WORKSPACE_DIR}}/DS-strategy/inbox/fleeting-notes.md`

3. **Реестры:**
   - `{{WORKSPACE_DIR}}/DS-strategy/docs/WP-REGISTRY.md` — для регистрации
   - `{{WORKSPACE_DIR}}/DS-strategy/current/WeekPlan W*.md` — для обновления бюджета
   - `memory/MEMORY.md` — для текущих РП

## Алгоритм

### 1. Разобрать предложение из WeekPlan

- Найди в конце WeekPlan последнее предложение Note-Review с пометкой `Классификация: Задача`
- Извлеки:
  - **текст заметки** (сам intent задачи)
  - **контекст** (область, связанные системы)
  - **время появления** (из Note-Review timestamp)

### 2. Определить параметры РП

**Правило:** параметры определяет Стратег, опираясь на контекст и классификацию Note-Review. Это НЕ простое копирование текста.

| Параметр | Источник | Правило |
|----------|----------|--------|
| **name** | Текст заметки + контекст | Переформулировать как **артефакт-существительное** (не глагол). Примеры: ✅ «Health-отчёт агентов в Telegram», ❌ «Сделать health-отчёт». Если заметка абстрактна → уточнить на основе контекста. Max 80 символов |
| **area** | Контекст заметки (Note-Review указывает область) | Доменная область в формате kebab-case: `exocortex-health`, `warehouse-automation`, `finance-reporting`. Если область неясна → спросить пользователя или выбрать на основе цепочки: заметка → WeekPlan контекст → Dissatisfactions → Strategy |
| **repo** | По умолчанию | `DS-strategy` (по умолчанию для личных РП). Если в контексте упомянут другой репо → использовать его |
| **budget** | Оценка на основе сложности | Оценить в часах: 0.5h, 1-1.5h, 2-2.5h, 3-4h. Простые задачи (проверка, быстрый фикс) → 0.5h. Стандартные (реализация одного компонента) → 1-2h. Комплексные (архитектура, интеграция) → 3h+. При неясности → 1-1.5h |
| **criteria** → ✅ строки | Из intent заметки, может быть несколько | Формулировать как проверяемые факты (не процесс): ✅ «Отчёт отправляется в Telegram каждый час», ❌ «Расширить Telegram-бот». Каждый критерий = одна строка с ✅ |

### 3. Проверить бюджет

- Прочитай текущий WeekPlan → найди сумму бюджетов активных РП (in_progress + pending)
- Если новый бюджет + существующие > ~16h недели → **предупреди** в комментарии WP-файла:
  ```
  <!-- ВНИМАНИЕ: Бюджет недели может быть перегружен. Рекомендация: найти РП для переноса на следующую неделю или уточнить бюджеты -->
  ```
  Но РП создаёшь в любом случае (не блокируешь).

### 4. Создать WP-файл в inbox/

**Путь:** `{{WORKSPACE_DIR}}/DS-strategy/inbox/WP-NNN-{slug}.md`

**Как найти NNN:**
- Прочитай WP-REGISTRY.md → найди максимальный номер (сейчас, вероятно, WP-131)
- Новый номер = max + 1

**Формат файла:** YAML frontmatter + markdown

```yaml
---
wp: {NNN}
title: {name}
area: {area}
repo: {repo}
budget: {budget}
priority: High  # по умолчанию для новых из Note-Review
status: pending
criteria:
  - ✅ {критерий 1}
  - ✅ {критерий 2}
  - ✅ {критерий 3}
---

# WP-{NNN}: {name}

> **Контекст:** {2-3 предложения о проблеме}
> **Цель:** {одна строка о результате}

## План

- [ ] {шаг 1}
- [ ] {шаг 2}
- [ ] {шаг 3}

## Статус

**Created:** YYYY-MM-DD (Note-Review → заметка)
**Assigned to:** Strategist (открытие), Claude/Codex (реализация)
```

### 5. Зарегистрировать в 5 местах

#### 5.1. WP-REGISTRY.md

Добавь новую строку в начало таблицы (после header):
```
| {NNN} | {name} | {repo} | {budget} | pending | High | {area} |
```

#### 5.2. WeekPlan W{N}

Найди текущую неделю в `{{WORKSPACE_DIR}}/DS-strategy/current/`. В таблице РП добавь строку (в раздел pending):
```
| 🟡 | {NNN} | **{name}** | {budget} | pending | {area} |
```

Обнови итоговый бюджет в секции.

#### 5.3. MEMORY.md («РП текущей недели»)

Добавь в таблицу:
```
| {NNN} | {name} | {budget} | pending | {дедлайн или YYYY-MM-DD + 5 дней} |
```

#### 5.4. fleeting-notes.md

Найди исходную заметку (она помечена как `Заголовок` или `**Заголовок**` без bold).
Добавь в конец описания ссылку:
```
[Converted to WP-{NNN}](../inbox/WP-{NNN}-{slug}.md)
```

#### 5.5. WeekPlan-предложение

Найди в конце WeekPlan строку Note-Review:
```
> **Note-Review:** ... Классификация: Задача ...
```

Замени на:
```
> ✅ **Note-Review → WP-{NNN}:** Конвертировано [время]
```

(Так пользователь видит, что предложение обработано)

### 6. Завершить

- Закоммить все изменения:
  ```
  git add {{WORKSPACE_DIR}}/DS-strategy/
  git commit -m "feat: WP-{NNN} (Note-to-WP) — {name}"
  git push
  ```

- Вывести отчёт:
  ```
  ✅ WP-{NNN} создан
  📍 Файл: inbox/WP-{NNN}-{slug}.md
  📊 Зарегистрирован: REGISTRY + WeekPlan + MEMORY + fleeting-notes
  💾 Коммит: [hash]
  ```

## Исключения и обработка ошибок

| Ситуация | Действие |
|----------|----------|
| Предложение Note-Review неполное (нет контекста) | Запросить уточнение пользователя. Не создавать WP с пустыми параметрами |
| Дублирование (похожий РП уже существует) | Проверить REGISTRY. Если intent одинаков → спросить: «Это продолжение WP-N или новый РП?». Если продолжение → добавить в контекст существующего РП, новый не создавать |
| Неясная область (area) | Спросить пользователя: «Какая область: exocortex-*, warehouse-*, finance-* и т.д.?» Не создавать с area=unknown |
| Бюджет не определяется | По умолчанию 1-1.5h. Пользователь может скорректировать после создания |
| Нет WeekPlan текущей недели | Поиск в archive/ или создание нового (если это воскресенье вечер — заранее) |
